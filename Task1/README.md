# Задание 1. Проектирование технологической архитектуры

### 1.Определить стратегию масштабирования и отказоустойчивости.

Для данной системы подойдет горизонтальное масштабирование, которая будет более эффективнее по сравнению с вертикальной. Так, как у компании, уже внедрен Kubernetes, будем от этого оталкиваться.

1. **Horizontal Pod Autoscaler (HPA)**

🔹 **Проблемы**

- Core-приложение и API подвержены скачкам нагрузки (до 50 RPS на поиск и 250 RPS от партнёров).

✅ **Решение**

- Настроить HPA для всех stateless-сервисов (core-app, ins-product-aggregator, client-info).
- Метрики: начать с CPU/Memory, затем подключить кастомные метрики через Prometheus Adapter (например, RPS или latency)
- Установить разумные границы (minReplicas, maxReplicas)

2. **Cluster Autoscaler**

🔹 **Проблемы**

- Простой HPA не решит проблему, если в кластере физически не хватает ресурсов (например, нет места под новые Pod'ы).
- При резких всплесках нагрузки (рекламная кампания, DDoS от партнёра, ночные пики в регионах) кластер должен уметь быстро расширяться.

✅ **Решение**

- Включить и настроить Cluster Autoscaler на уровне Kubernetes:
- Автоматическое добавление и удаление worker-нод в зависимости от потребности в ресурсах (например, если HPA хочет запустить 10 pod'ов, но нет места)

3. **Георезервирование**

🔹 **Проблемы**

- Сейчас весь кластер размещён в одной зоне доступности, что создаёт единую точку отказа.
  При сбое в дата-центре или зоне — приложение становится полностью недоступным.
- У пользователей из отдалённых регионов (например, Дальний Восток, Сибирь) страницы грузятся медленно, а API-ответы идут с высокой латентностью.
- Согласно бизнес-целям, InsureTech планирует развитие в разных часовых поясах России, а это требует устойчивости к региональным сбоям и равномерной производительности по всей стране.

✅ **Решение**

-Развернуть второй (или третий) Kubernetes-кластер в другом регионе (например, Москва + Новосибирск или Москва + Владивосток)

- Возможно, нужно использовать публичные облака, которые поддерживают мультизональность по РФ.
- Внедрить GSLB (Global Server Load Balancer). Пользователь попадает в ближайший к нему регион (по гео-IP).В случае недоступности одного региона — происходит failover на другой.
- Использовать Patroni для построения отказоустойчивых кластеров

4. **Мониторинг**

🔹 **Проблемы**

- Без метрик и алертов нельзя понять, когда кластеру не хватает ресурсов, или HPA не масштабируется как ожидалось.

- Проблемы выявляются уже после деградации.

✅ **Решение**

- Установить и настроить Prometheus + Grafana + Alertmanager
- Создать дашборды по: нагрузке на pod’ы, узлы, работе HPA и CA, латентности, RPS
- Настроить алерты по ключевым SLA/SLO.

### 2. Развёртывание приложения в нескольких зонах доступности

**a. Конфигурация развёртывания приложения в Kubernetes**

✅ Решение: Используем независимые кластеры Kubernetes в каждой площадке (регионе), например:

- Москва (основной кластер)
- Новосибирск или Владивосток (вторичный кластер)

**Комментарий:**
Мы не используем растянутый кластер, потому что:

- Rастянутый кластер усложняет сетевую инфраструктуру (latency, etcd-синхронизация, отказоустойчивость control-plane);
- Проблемы одной зоны (например, нестабильная сеть) могут нарушить работу всего кластера;
- Независимые кластеры обеспечивают лучший контроль, локальную устойчивость и позволяют независимо масштабироваться.

**b. Балансировка нагрузки**

✅ **Решение**

- Используем GSLB (Global Server Load Balancing) через публичный DNS-провайдер (например, Cloudflare, AWS Route 53, Yandex.Cloud DNS), который поддерживает GeoDNS и health-check.
- GSLB по gео-IP определяет, куда направить пользователя (например, клиент из Сибири попадает в Новосибирск).
  Если регион недоступен — происходит failover на другой.
- Используем Ingress Controller, например NGINX внутри каждого региона

**с. Фейловер-стратегия**

✅ Решение: Активно-активный

- В случае падения одного региона Active-Active позволяет автоматически обрабатывать запросы в другом, без простоев и потерь доступности.
- Active-Passive предполагает переключение (failover), что влечёт время простоя и риски потери данных, чего InsureTech не может себе позволить при текущем уровне SLA. Должны быть минмиальные риски так, как доступность приложения должна быть равна 99,9%. RTO = 45 минут, RPO = 15 минут, Простыми словами, downtime у Active-Passive больше.

**d. Конфигурация базы данных (PostgreSQL)**

🎯 Цели:

- RTO ≤ 45 минут — сервис должен быть восстановлен в течение 45 минут после сбоя.
- PO ≤ 15 минут — потеря данных не должна превышать 15 минут.

✅ Решение:

- Используем кластер PostgreSQL с многорегиональной репликацией на основе Patroni
  > - Внутри региона — синхронная репликация (поддержание точной копии в пределах региона).
  > - Между регионами — асинхронная репликация (с минимальной задержкой, чтобы RPO ≤ 15 мин).
  > - Patroni с etcd или Consul обеспечивает автоматический failover и согласованное лидерство.
- Health-check и мониторинг
- Балансировка трафика на чтение/запись. Используем PgBouncer или HAProxy с read/write split

### 3. Шардирование БД

✅ Решение: На текущем этапе — нет, но в будущем — потенциально да.

- Относительно низкая нагрузка. Максимальная зафиксированная нагрузка: 50 RPS (поиск) и 10 RPS (оформление), 250 RPS по API — это не критично для одной масштабируемой PostgreSQL-системы
- Шардирование добавит сложности в логике доступа, особенно с учётом того, что клиентские и страховые данные могут потребоваться совместно
- Нет естественных границ для шардинга, вроде разных географий, бизнес-юнитов или продуктовых направлений
- PostgreSQL масштабируется вертикально и через реплики
