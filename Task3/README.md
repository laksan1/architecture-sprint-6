# 🧠 Задание 3. Переход на Event-Driven архитектуру

---

## 1. 📈 Архитектурные проблемы и риски InsureTech

### 🔥 Производительность и масштабирование

- core-app — монолит, сложно масштабировать независимо.
- ins-product-aggregator делает синхронные вызовы ко всем страховым при каждом запросе → увеличит задержки и риски таймаутов при росте числа страховых (сейчас 5, будет 10).
- Локальные реплики тарифов (в core-app и ins-comp-settlement) синхронизируются по расписанию (раз в 15 минут и раз в сутки соответственно) — неустойчивая и несогласованная модель.
- ins-comp-settlement зависит от core-app для получения данных о страховках → tight coupling, рост трафика ночью может привести к сбоям.

### ⚠️ Надёжность и отказоустойчивость

- Ошибки сторонних API (страховых) напрямую влияют на скорость и успех оформления страховок.
- Задержки или падения ins-product-aggregator тормозят весь процесс оформления в core-app.
- Все сервисы и БД в одной зоне доступности — отсутствие георезервирования.
- Отсутствие централизованного мониторинга и алертинга → позднее реагирование на сбои.

### 🚧 Нарушение SLA

- Один партнёр способен перегрузить весь API (достигал 250 RPS) — нет ограничения per-client (rate limiting).
- Отсутствие изоляции B2B и B2C потоков — ухудшает опыт обычных пользователей.

### 📈 Риски при росте числа страховых партнёров

- Удвоение числа страховых компаний пропорционально увеличит нагрузку на ins-product-aggregator.
- Рост ошибок и таймаутов при агрегировании, особенно в пиковые часы.

### 🔐 Безопасность и стабильность API

- Публичный доступ к API через LoadBalancer Kubernetes без ограничений или защиты от злоупотреблений.
- Нет throttling или circuit breaker'ов для внешних вызовов и внутренних зависимостей.

### 🧩 Поддержка и эволюция системы

- Жёсткие зависимости между сервисами усложняют внесение изменений.
- core-app — узкое место и bottleneck для большинства бизнес-процессов.

## 2. 📊 План действий

### ⚡ Переход на Event-Driven взаимодействие

- Перейти от синхронных запросов `core-app` и `ins-comp-settlement` к `ins-product-aggregator` на event-streaming:

  - `ins-product-aggregator` публикует события `ProductCatalogUpdated` при изменении данных страховых.
  - `core-app` и `ins-comp-settlement` подписываются на эти события и обновляют локальные реплики.

- Использовать **Transactional Outbox** в `ins-product-aggregator`:

  - Обеспечит надёжную публикацию событий `ProductCatalogUpdated` в Kafka.
  - Исключает риск рассинхронизации между БД и шиной событий.

- Заменить ночной синхронный запрос от `ins-comp-settlement` к `core-app` на асинхронную обработку:
  - `core-app` публикует событие `InsurancePolicyCreated`.
  - `ins-comp-settlement` подписывается и накапливает данные в своей БД для расчётов.

---

### 🧠 Надёжность и масштабирование

- Ввести retry-механику и DLQ (Dead Letter Queue) на стороне подписчиков событий.
- Использовать распределённый кеш (например, Redis) для хранения актуальных агрегированных предложений, если отклик по API страховых всё же нужен в real-time.

---

### 🧪 Мониторинг и SLA

- Добавить алерты на превышение RPS по партнёрам и задержки ответов от внешних API.
- Внедрить трассировку (например, OpenTelemetry) и централизованный логинг (ELK, Loki).

---

### 🌐 Инфраструктура

- Разнести PostgreSQL на мастер-реплику с репликацией и резервным экземпляром.
- Перевести базу в managed-сервис с автоматическим failover.

---

### 📦 Использование паттерна Transactional Outbox

- **Да, паттерн Transactional Outbox будет использоваться.**
- Он необходим для гарантированной доставки событий (например, `ProductCatalogUpdated`, `InsurancePolicyCreated`) при обновлении данных в БД.
- Это исключает несогласованность между базой и шиной событий при сбоях или откатах транзакций.
- Реализация:

  - Событие записывается в отдельную таблицу outbox в рамках основной транзакции.
  - Фоновый процесс или Debezium (CDC) отслеживает и публикует события в Kafka

- `ins-product-aggregator` — если изменения касаются данных о продуктах (например, обновление списка страховых продуктов). Если сервис агрегирует данные от страховых компаний и это изменение важно для других микросервисов (например, для core-app и ins-comp-settlement), то события об этих изменениях должны быть записаны в outbox именно в ins-product-aggregator.

- `core-app` — если изменения касаются информации о страховках (например, новый заказ, изменение данных клиента). Если изменения происходят в core-app (например, добавление нового страхового полиса или его обновление), то события об этих изменениях должны быть записаны в outbox именно в core-app.

- `ins-comp-settlement` — если изменения касаются расчетов с партнёрами, например, информация о расчёте комиссий для страховых компаний, то здесь также можно использовать Transactional Outbox для отправки событий о завершении расчётов и обновления данных в других сервисах.
